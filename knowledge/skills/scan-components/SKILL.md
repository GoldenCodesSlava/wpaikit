# Scan Components

## Purpose

Build `.wpaikit/components-registry.md` — a machine-readable index of all existing components and classes.
This file is read by `/figma-to-block` and `/design-system-to-code` before generating code so they can reuse what already exists instead of duplicating it.

## Phase 1 — Locate SCSS entry file

Search for the main SCSS entry file in this order:
1. `frontend/src/css/main.scss`
2. `frontend/src/main.scss`
3. `frontend/src/index.scss`
4. `frontend/src/app.scss`

If none found → stop: "Cannot find main SCSS entry file. Run from `wp-content/themes/boilerplate/`"

Read the entry file. Parse every `@import`, `@use`, and `@forward` line.
Resolve each to an absolute path relative to the entry file's directory.

Build `SCSS_FILES` — the full list of imported SCSS file paths.

Example:
```
frontend/src/css/base/_typography.scss
frontend/src/css/base/_reset.scss
frontend/src/css/components/_button.scss
frontend/src/css/components/_badge.scss
frontend/src/css/blocks/_hero-block.scss
```

Skip vendor imports (paths starting with `~` or `node_modules`).

## Phase 2 — Extract SCSS classes

For each file in `SCSS_FILES`, read its content and extract:

### BEM root classes

Find all top-level class selectors (`.class-name {`). These are the BEM blocks.

### Modifiers and elements

For each BEM root, find nested `&--modifier` and `&__element` selectors.

Map full names:
- `&--primary` inside `.btn` → `.btn--primary`
- `&__icon` inside `.btn` → `.btn__icon`

### Build SCSS_REGISTRY per file

```
File: frontend/src/css/components/_button.scss
Block: .btn
  Elements: __icon, __label
  Modifiers: --primary, --secondary, --ghost, --sm, --md, --lg, --disabled
```

Skip files with no class definitions (reset, variables, mixins).

## Phase 3 — Scan Twig files

Scan two directories:
- `views/components/` — reusable UI components
- `views/partials/` — layout-level partials (header, footer, nav)

For each `.twig` file found, read it and extract:

### Include path

Relative to theme root: `views/components/button.twig`

### Params from header comment

Every generated Twig file has a header comment in this format:
```twig
{#
  Component: Button
  Variants:
    - type: Primary | Secondary | Ghost  (default: Primary)
    - size: SM | MD | LG                 (default: MD)
  Usage: {% include 'views/components/button.twig' with { type: 'Primary', size: 'MD', label: '', url: '' } only %}
#}
```

Extract:
- Component/Partial name
- Variants/params with their options and defaults
- Usage line (the `{% include %}` snippet)

If no header comment exists → record path only, mark params as `unknown`.

### Build TWIG_REGISTRY per file

```
File: views/components/button.twig
Type: component
Params: type (Primary|Secondary|Ghost, default: Primary), size (SM|MD|LG, default: MD), disabled (bool, default: false)
Include: {% include 'views/components/button.twig' with { type: 'Primary', size: 'MD', label: '', url: '' } only %}
```

## Phase 4 — Write registry

Write `.wpaikit/components-registry.md`.

If file already exists → overwrite it.

### File format

```md
# Components Registry

<!-- Generated by /scan-components on {DATE} -->
<!-- Run /scan-components to update after adding new components -->

## How to use

Before generating new code, check this registry.
- If a component exists → use `{% include %}` instead of writing new markup
- If a BEM class exists → extend it with a modifier instead of creating a new block

---

## Twig Components

{for each file in views/components/}

### {ComponentName}

- **Path:** `views/components/{name}.twig`
- **Params:** {param list}
- **Include:**
  ```twig
  {% include 'views/components/{name}.twig' with { ... } only %}
  ```

{end for}

---

## Twig Partials

{for each file in views/partials/}

### {PartialName}

- **Path:** `views/partials/{name}.twig`
- **Params:** {param list or "none / unknown"}

{end for}

---

## SCSS Components

{for each component file in SCSS_FILES}

### .{block-name}

- **File:** `{file path}`
- **Elements:** {__element list or "none"}
- **Modifiers:** {--modifier list or "none"}

{end for}

---

## SCSS Base

{for each base file in SCSS_FILES with classes}

### {filename}

- **File:** `{file path}`
- **Classes:** {class list}

{end for}
```

## Phase 5 — Report

```
=== Components Registry ===

Scanned:
  SCSS files:  N (from main.scss imports)
  Twig components: N (views/components/)
  Twig partials:   N (views/partials/)

Registry written:
  .wpaikit/components-registry.md

Summary:
  Components:  Button, Badge, Card, ...
  Partials:    Header, Footer, ...
  SCSS blocks: .btn, .badge, .card, ...

Next steps:
  Run /figma-to-block — it will now read the registry and reuse existing components.
  Re-run /scan-components after adding new components.
```

## Boundaries

- Read-only scan — never modify SCSS or Twig files
- Never modify `design-system.json`
- Only writes `.wpaikit/components-registry.md`
- If a Twig file has no header comment, record it with `params: unknown` — do not skip it
