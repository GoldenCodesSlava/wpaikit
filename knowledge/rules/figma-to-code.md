# Figma to Code — Universal Rules

These rules apply to every command that translates Figma designs into code:
`/figma-to-block`, `/design-system-to-code`, and any future generation command.

Read this file before generating any Twig, SCSS, PHP, or config file from a Figma design.

---

## 1. Never write Tailwind utilities in Twig or PHP

All visual styling lives in SCSS. Twig uses semantic BEM class names only.

**Wrong:**
```twig
<section class="flex flex-col md:flex-row gap-6 py-12 bg-brand-primary">
```

**Correct:**
```twig
<section class="hero-block">
```

```scss
.hero-block {
  @apply flex flex-col gap-6 py-12 bg-brand-primary;

  @screen md {
    @apply flex-row;
  }
}
```

---

## 2. One SCSS partial per component or block

| Type | Twig path | SCSS path |
|---|---|---|
| Component | `views/components/button.twig` | `frontend/src/components/_button.scss` |
| Block | `views/blocks/HeroBlock/HeroBlock.twig` | `frontend/src/blocks/_hero-block.scss` |

Every generated SCSS partial must be imported in the main SCSS entry file (`frontend/src/main.scss` or equivalent).

---

## 3. BEM naming

```
.block-name               ← block
.block-name__element      ← element
.block-name--modifier     ← modifier
```

Examples:
```scss
.btn { }
.btn__icon { }
.btn--primary { }
.btn--sm { }
.btn--disabled { }

.hero-block { }
.hero-block__content { }
.hero-block__heading { }
.hero-block__cta { }
```

Use kebab-case. Match the Figma layer name converted to kebab-case.

---

## 4. Mobile-first responsive

Base classes = mobile. Larger breakpoints use Tailwind screen prefixes inside `@apply`:

```scss
.hero-block {
  @apply flex flex-col gap-4 py-8 px-4;    // mobile

  @screen md {
    @apply flex-row gap-12 py-24 px-16;   // desktop
  }
}
```

Always generate responsive styles — even if only a desktop frame was provided in Figma.
Use smart defaults for mobile when no mobile frame exists (see skill for defaults table).

---

## 5. Use design token names, not raw values

Always reference token names from `design-system.json`, never hardcode hex values or raw px.

```scss
// Wrong
color: #0f172a;
padding: 48px;

// Correct
@apply text-primary py-12;
```

Token → Tailwind class mapping:
- `Colors/brand/primary` → `brand-primary` (as Tailwind color)
- `Colors/text/primary` → `text-primary`
- `Colors/bg/canvas` → `bg-canvas`
- `Spacing/N` → Tailwind spacing scale (`Spacing/4` → spacing-4 → `p-4`, `gap-4`, etc.)
- `Radius/lg` → `rounded-lg`
- `Heading/4xl` → `text-4xl`

---

## 6. Variant classes as BEM modifiers

For components with variants (Button Type, Badge Color, etc.), map each variant to a BEM modifier:

```scss
.btn {
  @apply inline-flex items-center font-medium transition-colors;  // base

  // Type variants
  &--primary   { @apply bg-brand-primary text-inverse; }
  &--secondary { @apply bg-brand-accent text-inverse; }
  &--ghost     { @apply border border-brand-primary text-primary bg-transparent; }

  // Size variants
  &--sm { @apply text-sm px-3 py-1.5 rounded-md; }
  &--md { @apply text-base px-5 py-2.5 rounded-lg; }
  &--lg { @apply text-lg px-6 py-3 rounded-lg; }

  // State variants
  &--disabled { @apply opacity-40 pointer-events-none; }
}
```

In Twig, combine the modifiers:
```twig
<a class="btn btn--{{ type|lower }} btn--{{ size|lower }} {% if disabled %}btn--disabled{% endif %}">
```

## 7. File header comment

Every generated Twig file starts with a comment block:

```twig
{#
  Component: Button
  Variants:
    - type: Primary | Secondary | Ghost
    - size: SM | MD | LG
  Usage: {% include 'views/components/button.twig' with { type: 'Primary', size: 'MD', label: 'Click', url: cta.url } %}
#}
```

Every generated SCSS partial starts with:

```scss
// Component: Button
// Variants: type (Primary, Secondary, Ghost), size (SM, MD, LG)
// Generated by /design-system-to-code
```

## 8. Container pattern for blocks

Most blocks constrain their content width using Tailwind's `container` class.

**Standard structure (container = yes):**
```twig
<section class="hero-block">
  <div class="hero-block__inner container">
    <div class="hero-block__content">...</div>
  </div>
</section>
```

```scss
.hero-block {
  @apply py-8;         // vertical padding + bg only — no px

  @screen md {
    @apply py-24;
  }

  &__inner { }         // intentionally empty — container handles width/centering
}
```

Rules:
- `section` = vertical padding (`py-*`) + background only — never `px-*`
- `__inner container` = width constraint and centering via Tailwind `container`
- `container` is a structural layout primitive — it is allowed directly in Twig as an exception to the no-utility rule
- Full-width blocks (edge-to-edge images, maps) skip the `__inner container` wrapper

## 9. ACF field groups — tab structure

Every block's ACF field group uses tabs to organise fields. Tabs are `type: "tab"` entries in the flat `fields` array, placed before the fields they group.

**Tab order (always this sequence):**

| Tab | Required | Fields |
|---|---|---|
| **Settings** | Always | `is_active` (true_false), `block_custom_id` (text) |
| **Content** | If content fields exist | text, textarea, wysiwyg, link |
| **Media** | If media fields exist | image, video, file |
| **Appearance** | If appearance fields exist | select, radio, color_picker, button_group |
| **[Repeater name]** | One per repeater | The repeater field + sub-fields (flat, no inner tabs) |

**Rules:**
- Settings tab is the only tab that is always generated — even if the block has no other fields
- All other tabs are created only when they have at least one field
- Each repeater gets its own tab named after the repeater (e.g. `Items`, `Team Members`)
- Sub-fields inside a repeater are flat — no nested tabs
- `endpoint: 1` on the last tab in the group; `endpoint: 0` on all others
- Standard Settings fields (`is_active`, `block_custom_id`) are added automatically — never omit them

## 10. Per-file validation loop

**Every generated or modified file must go through this loop before moving to the next file:**

```
1. Write the file
2. Validate against the checklist below
3. Fix all violations found
4. Re-validate — only proceed when all checks pass
```

This applies to every SCSS partial, every Twig file, every PHP class, and every ACF JSON file.

### SCSS checklist

- [ ] Header comment present (`// Component: Name` or `// Block: Name`)
- [ ] All class names are BEM kebab-case — no camelCase, no utility classes as class definitions
- [ ] No Tailwind utilities written as raw CSS properties — only `@apply`
- [ ] No hardcoded hex colors anywhere — not in `@apply`, not in raw CSS properties
- [ ] If a color value is needed but has no matching entry in `tailwind.config.js`:
  1. Stop and show the user: `Color #xxxxxx not found in tailwind.config.js. Add as [proposed-name]? [yes / rename / skip]`
  2. On `yes` or `rename` → add to `theme.extend.colors` in `tailwind.config.js` AND add to `design-system.json → colors`
  3. On `skip` → use the closest existing token instead, note the substitution
  4. Never proceed with a raw hex in the output file
- [ ] Mobile-first: base classes = mobile, `@screen md {}` = desktop
- [ ] `px-*` not applied to outer `section` when container is used — vertical only (`py-*`)
- [ ] `&__inner {}` is empty when Tailwind `container` handles width (comment explains why)
- [ ] Partial is imported in `main.scss` (or equivalent entry file)

### Twig checklist

- [ ] Header comment present (`{# Component/Block: Name ... #}`)
- [ ] No Tailwind utility classes in `class=""` attributes — only BEM class names
- [ ] Exception: `container` on `__inner` div is allowed
- [ ] Plain text fields use `{{ field }}` — no `|raw`
- [ ] URLs use `{{ field.url|raw }}`
- [ ] Wysiwyg content uses `{{ field|raw }}`
- [ ] Every optional field wrapped in `{% if field %}...{% endif %}`
- [ ] No PHP logic or expressions inside Twig
- [ ] Known components from `.wpaikit/components-registry.md` use `{% include %}` — never re-implemented

### PHP checklist (blocks only)

- [ ] `declare(strict_types=1)` is the first statement
- [ ] Namespace is `Boilerplate\Theme\Blocks`
- [ ] All ACF fields fetched with `get_field()`
- [ ] String fields default to `''`, objects/arrays default to `null` or `[]`
- [ ] Image fields wrapped in `new \Timber\Image()`
- [ ] No HTML output in PHP — all markup in Twig

### ACF JSON checklist (blocks only)

- [ ] Settings tab always present and first, with `is_active` and `block_custom_id`
- [ ] Tabs created only for field types that exist
- [ ] Last tab has `endpoint: 1`, all others have `endpoint: 0`
- [ ] `new_tab_group` is never `true` on any tab — always `false` or omitted
- [ ] Field keys follow `field_{block_snake}_{field_name}` format
- [ ] `link` fields have `return_format: array`
- [ ] `image` fields have `return_format: id`
- [ ] `instructions` field contains plain text only — never HTML tags

## 11. Twig includes for components in block templates

When a block uses a design system component, use `{% include %}`:

```twig
{# In HeroBlock.twig #}
{% include 'views/components/button.twig' with {
  type: 'Primary',
  size: 'LG',
  label: cta.title,
  url: cta.url
} only %}
```

Never re-implement component markup inside block templates.
