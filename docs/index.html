01

Установка wpaikit

Начинайте с проверки окружения. CLI ставится глобально и дальше используется для создания WordPress-проекта, установки knowledge base и обновления команд.

Минимальные требования

Node.js 18+

PHP 8.1+

Git, Composer, pnpm

Python + fonttools для команды /setup-fonts

Глобальная установка

npm install -g @veaceslav-golden/wp-ai-kit-core
npm install -g @veaceslav-golden/wp-ai-kit

Проверка

wpaikit --version
wpaikit doctor
Что делает doctor Проверяет доступность Node, Git, Composer, npm и права записи в текущей директории. Если чего-то не хватает, команда показывает, что надо установить до старта проекта.
2
CLI

wpaikit init

Главная стартовая команда. Она скачивает WordPress, подключает boilerplate, переименовывает тему и готовит проект к ручной настройке базы данных.

Command wpaikit init

Интерактивные вопросы

Project name — человекочитаемое имя проекта.

Slug — kebab-case для папки темы.

Namespace — PHP namespace в PascalCase.

Location — новый подкаталог или текущая директория.

Автоматические шаги

  • Скачивает последнюю стабильную WordPress-сборку.
  • Клонирует boilerplate в wp-content/.
  • Переименовывает тему, namespace и text domain.
  • Запускает composer install и frontend build.
  • Создает .wpaikit.json с метаданными проекта.

Полезные flags

wpaikit init --name my-project

wpaikit init --name my-project --yes

wpaikit init --name my-project --dry-run

wpaikit init --name my-project --skip-install

Ручные шаги после init

  1. Создайте локальную базу данных через Herd, MAMP, TablePlus или CLI.
  2. Заполните доступы в wp-config.php.
  3. Откройте сайт в браузере и пройдите WordPress wizard.
  4. Активируйте тему в wp-admin -> Appearance -> Themes. 
define('DB_NAME', 'salon_frumusete');
define('DB_USER', 'root');
define('DB_PASSWORD', '');
define('DB_HOST', 'localhost');
define('BOILERPLATE_DEV', true);

AI setup

Установка Knowledge Base

После создания проекта перейдите в корень темы. Все AI-команды ниже должны запускаться именно оттуда, чтобы агенты видели правильные rules, prompts и skills.

Theme root cd wp-content/themes/{slug} wpaikit knowledge install

Что копируется

knowledge/ — rules, prompts, skills.

CLAUDE.md — инструкции для Claude Code.

AGENTS.md — инструкции для Codex/OpenAI.

.claude/commands/ — slash-команды.

Когда обновлять

После обновления глобального CLI или когда команда получила новый prompt/skill. 

npm install -g @veaceslav-golden/wp-ai-kit-core@latest
npm install -g @veaceslav-golden/wp-ai-kit@latest
wpaikit knowledge install

Рабочее правило

CLI делает детерминированные операции. AI-команды интерпретируют Figma, brief, структуру блоков и решения по дизайну.

04

Команды и когда их запускать

Команды запускаются в Claude Code из корня темы. Каждая команда предварительно читает нужные файлы из knowledge/.

Design

/generate-design

Интерактивно собирает brief, Figma project URL, референсы, Design System, brand assets, icon library и Pexels API key, затем генерирует UI в Figma.

Audit

/design-quality-check

Проверяет brief или Figma frames на category reflex, banned patterns, цветовую стратегию, register consistency, theme logic, typography и layout rhythm.

Audit

/analyze-figma

Аудитирует Figma frame на готовность к реализации: Auto Layout, токены, responsive variants, reusable components и semantic naming.

Tokens

/figma-design-system

Извлекает colors, typography, spacing, radius и UI patterns из 4-6 representative frames. Создает Design System page, Variables, Text Styles и .wpaikit/design-system.json.

Figma

/figma-components

Создает Figma Component Sets с вариантами Type x Size x State на основе patterns из design-system.json.

Figma

/figma-update-design-system

Добавляет новые frames в существующий Design System без удаления старых токенов. Делает diff, append в Figma и merge JSON.

Figma

/figma-sync-tokens

Синхронизирует design-system.json, когда дизайнер вручную меняет Variables или Text Styles. Удаленные токены помечаются deprecated.

Prep

/prep-figma

Создает страницу Dev Ready, семантически переименовывает layers, связывает styles с tokens, заменяет detached элементы компонентами и добавляет Auto Layout.

Fonts

/setup-fonts

Сравнивает нужные font families/weights с файлами в frontend/src/fonts/, удаляет лишнее, конвертирует TTF/OTF в woff2/woff и генерирует _fonts.scss.

Code

/design-system-to-code

Генерирует Twig components, SCSS partials, обновляет Tailwind config и добавляет imports в main.scss.

Code

/scan-components

Индексирует Twig и SCSS компоненты, BEM blocks/elements/modifiers и пишет .wpaikit/components-registry.md для повторного использования.

Code

/figma-to-block

Генерирует один ACF block за запуск: PHP class, Twig template, SCSS partial и ACF JSON. После каждого блока надо sync в wp-admin Custom Fields.

QA

/validate-code

Проверяет SCSS, Twig и PHP по rules. Может чинить BEM, raw CSS, hardcoded hex, autoescape mistakes, namespaces и image fields.

Reference

/get-comment-for-frontend

Читает базу WordPress напрямую, парсит ACF blocks из post_content и генерирует developer reference с путями Twig, SCSS, PHP и JS.

05

Полные workflow

Два базовых сценария: проект с готовым дизайном и проект, где дизайн сначала генерируется в Figma.

Workflow A — дизайн уже есть в Figma

npm install -g @veaceslav-golden/wp-ai-kit-core
npm install -g @veaceslav-golden/wp-ai-kit
wpaikit doctor
wpaikit init

# manual:
# create DB, configure wp-config.php,
# complete WordPress wizard, activate theme

cd wp-content/themes/{slug}
wpaikit knowledge install

/analyze-figma <url>
/figma-design-system <url>
/figma-components
/prep-figma <url>
/setup-fonts
/design-system-to-code
cd frontend && npm run build
/scan-components
/figma-to-block <url>
/validate-code

Workflow B — проекта без дизайна

npm install -g @veaceslav-golden/wp-ai-kit-core
npm install -g @veaceslav-golden/wp-ai-kit
wpaikit doctor
wpaikit init
cd wp-content/themes/{slug}
wpaikit knowledge install

/generate-design
/design-quality-check <figma-url>
/figma-design-system <url>
/figma-components
/prep-figma <url>

# дальше как Workflow A:
/setup-fonts
/design-system-to-code
/scan-components
/figma-to-block <url>
/validate-code
Важно Один запуск /figma-to-block генерирует один блок. Для каждой секции страницы повторяйте команду отдельно и после каждого блока синхронизируйте ACF JSON в WordPress admin.

Частые ситуации

Короткая карта решений для ежедневной работы с проектом.

Ситуация Что запускать Комментарий
Новый блок из Figma /prep-figma -> /figma-to-block Prep можно пропустить только если frame уже dev-ready.
Дизайнер поменял цвета или шрифты /figma-sync-tokens После этого проверьте Tailwind и build.
В Figma добавились страницы /figma-update-design-system Новые токены добавляются, старые не удаляются.
Добавлена новая Twig/SCSS компонента /scan-components Обновляет registry для будущей генерации блоков.
Нужна проверка после генерации /validate-code Проверяет SCSS, Twig и PHP против rules.
Разработчику надо найти файлы страницы /get-comment-for-frontend <url> Пишет markdown-референс по блокам страницы.

Ключевые файлы проекта

Эти файлы и папки чаще всего нужны при разработке и отладке.

Metadata и AI

.wpaikit.json — имя, namespace, preset.

.wpaikit/design-system.json — cache токенов.

.wpaikit/components-registry.md — индекс компонентов.

Frontend

frontend/src/css/main.scss — SCSS entry.

frontend/tailwind.config.js — Tailwind tokens.

frontend/src/css/components/ — component partials.

frontend/src/css/blocks/ — block partials.

WordPress theme

blocks/ — PHP classes for ACF blocks.

views/blocks/ — Twig block templates.

views/components/ — reusable Twig components.

acf-json/ — ACF field groups to sync in admin.

FG

Figma -> ACF block: подробные этапы

Ниже остается прежняя детальная карта Figma-to-block процесса. Она полезна, когда проект уже инициализирован и нужно работать именно с дизайном и блоками.

00

Overview

The process is split into stages so each skill has one clear responsibility.

Active skills 13

Design generation, analysis, design system, prep, components, sync, fonts, code output, validation, block generation, and developer reference.

Primary input Figma URL

Frame, page-level frame, or multi-selection depending on the stage.

Code target Theme root

wp-content/themes/boilerplate/ or a renamed project theme.

Core rule: One top-level Figma frame should map to one ACF block. If the design structure does not reflect the future HTML structure, run analysis and preparation first.
1
Foundation

Design

The designer prepares the layout so an agent can read it safely.

What Figma should contain

  • Frames are named after blocks: HeroBlock, TeamBlock.
  • Layers use semantic names: Heading, CTA Button.
  • Structural layers use Auto Layout.
  • Colors, spacing, radius, and typography come from Variables or Styles.
  • Repeated elements are built as Figma Components.

4
Active

Prep Figma

Prepares a page-level frame for generation while preserving the visual result.

What can change

  • Layer naming and block boundaries.
  • Token bindings for colors, spacing, radius, and typography.
  • Auto Layout when geometry does not shift.
  • Component instance replacement without size changes.
Guardrail If a change may shift pixels, skip it and add it to the report.

Active

Components

Repeated UI patterns become Figma Components and then Twig includes.

Rules

No Tailwind utilities in Twig. Twig uses semantic BEM classes only.
